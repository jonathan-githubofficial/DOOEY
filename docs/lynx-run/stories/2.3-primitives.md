SPEC:

1. Lynx element/event facts this unit relies on (cite, do not re-derive):
   - No `<button>` element exists. Interactive containers are `<view bindtap={handler}
     catchtap={handler}>` - `bindtap` bubbles, `catchtap` stops propagation. Event props use
     Lynx's native `bind*`/`catch*` prefix convention, NOT React DOM's `onClick`:
     https://lynxjs.org/react/thinking-in-reactlynx.html ,
     https://lynxjs.org/api/react/document.directives
   - `<view>` has no `disabled` prop; gate interaction via `user-interaction-enabled={false}`
     or by simply not passing a handler (`bindtap={disabled ? undefined : onClick}`) plus a
     visual class change driven by the same boolean - no CSS selector involved.
   - `:active` IS a real, natively supported pseudo-class on Lynx
     (https://lynxjs.org/api/css/selectors.html) - unlike `:hover`/`:focus`/`:disabled`. Keep
     Tailwind's `active:` variant classes (e.g. `active:scale-95`) as-is; they compile straight
     through.
   - `<input>` supports a real `disabled` boolean prop directly, plus `bindinput` (value +
     selection), `bindfocus`/`bindblur` (each returns current value), `bindconfirm`,
     `bindselection`. No native two-way-binding sugar - drive `value` from React state via
     `bindinput`. https://lynxjs.org/api/elements/built-in/input
   - Tailwind class handling: `@lynx-js/tailwind-preset` filters/adapts standard utilities;
     `className` composition (hence `cn()`/`clsx`/`tailwind-merge`) works the same way as on
     any Tailwind project - nothing framework-specific to change in `cn.ts`.
     https://lynxjs.org/rspeedy/styling

     "inline-flex items-center justify-center gap-2 whitespace-nowrap rounded-full text-sm font-medium font-sans transition-colors active:scale-95",

6. Rewrite `src/components/ui/input.tsx`:
   ```tsx
   import { useState } from "react";
   import { cn } from "@/lib/cn";

   interface InputProps {
     className?: string;
     type?: "text" | "number" | "digit" | "password" | "tel" | "email";
     placeholder?: string;
     value?: string;
     disabled?: boolean;
     onInput?: (value: string) => void;
   }

   export function Input({ className, type, placeholder, value, disabled, onInput }: InputProps) {
     const [focused, setFocused] = useState(false);
     return (
       <input
         type={type}
         placeholder={placeholder}
         value={value}
         disabled={disabled}
         bindinput={(e: { detail: { value: string } }) => onInput?.(e.detail.value)}
         bindfocus={() => setFocused(true)}
         bindblur={() => setFocused(false)}
         className={cn(
           "flex h-10 w-full rounded-md border border-rule bg-transparent px-3 py-2 text-base text-ink font-sans",
           focused && "ring-2 ring-leaf",
           disabled && "opacity-50",
           className,
         )}
       />
     );
   }
   ```
   `focus-visible:ring-2 focus-visible:ring-leaf focus-visible:ring-offset-2` becomes a plain
   `focused && "ring-2 ring-leaf"` driven by `bindfocus`/`bindblur` state, since Lynx has no
   `:focus`/`:focus-visible` selector. `disabled` is Lynx `<input>`'s own real boolean prop
   (https://lynxjs.org/api/elements/built-in/input), passed straight through - no CSS-selector
   workaround needed there, only the opacity visual needs the JS-driven class.
