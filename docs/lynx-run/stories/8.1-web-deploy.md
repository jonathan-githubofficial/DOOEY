# 8.1 - Web deploy wiring

LAYER / TIER / RISK: L8-cutover / normal (sonnet) / MEDIUM (mechanical plumbing, but
a wrong host-shell path or env-var rename fails silently in production only - E2E
runs against the dev/test server shape, not the Docker image, so this unit must
verify the actual built artifact, not just trust that typecheck/build stay green).

GOAL: Rewire the Dockerfile's web-build stage (and confirm compose/deploy.yml need
no changes) so `docker build` / `docker compose up --build` produce the same
one-container image shape, now serving the Lynx (Rspeedy) web output instead of
the old Vite `dist/`.

CONTEXT (what exists when this unit starts):
- L1-L7 are committed and E2E-green through @l7. Unit 1.1 renamed `src` ->
  `src-legacy` (READ-ONLY reference) and scaffolded the ReactLynx app in a fresh
  `src/`; `npm run build` already runs the real `rspeedy build` (wired by 1.1),
  and `npm run typecheck` / `npm run lint` exist.
- Unit 1.5 built the E2E oracle. To run Playwright against "the Lynx web output"
  in headless Chromium at all, 1.5 necessarily already solved how to SERVE that
  output as a browsable page - that mechanism (whatever `e2e/README.md` and
  `playwright.config.ts`'s `webServer` describe) is the ground-truth reference for
  what this unit wires into Docker. Do not invent a second, different hosting
  mechanism.
- Unit 1.4 built the PB client seam (ported from `src-legacy/lib/pb.ts`), which
  replaced `import.meta.env.VITE_PB_URL` host resolution with whatever Lynx-side
  equivalent it uses (Rspeedy env vars are NOT Vite's `import.meta.env.VITE_*` -
  there is no forced prefix; 1.4 chose a real name).
- `src-legacy/` still holds whatever L1-L7 did not broom. The final broom is unit
  8.2, which runs AFTER this unit in this layer's unit order - do not wait for it
  and do not touch `src-legacy/` here; this unit only touches Dockerfile / compose
  / deploy.yml / the one `docs/deploy-google-cloud.md` line below.
- Lynx build model (verified 2026-07 against lynxjs.org/rspeedy/output and the
  integrate-with-existing-apps guide, since Lynx has little training-data
  coverage - re-verify by fetching if anything below looks off): `rspeedy build`
  emits to `dist/` by default. For a project with a `web` environment configured
  (`lynx.config.ts` / rspeedy config `environments: { web: {...} }`), the web
  artifact is a bundle file - typically `dist/main.web.bundle` (the `main` part
  follows the project's entry name; do not hardcode it, confirm on disk). Per the
  crib sheet, "Web output runs via @lynx-js/web-core / web-elements host" - the
  bundle is NOT a self-contained static site like Vite's `dist/` was. It must be
  loaded by a small host page that imports `@lynx-js/web-core` (`/client`) and
  `@lynx-js/web-elements`, and renders `<lynx-view url="...">`. That host page is
  either (a) emitted by the SAME rspeedy build if 1.1/1.5 configured the web
  environment to self-host, or (b) a small separate project (e.g. under `e2e/` or
  a `web-shell/` dir) that 1.5 built to view the bundle in Chromium. Find out
  which is actually true in THIS repo before editing the Dockerfile (step 1).

SPEC (numbered, concrete):
1. Read `e2e/README.md` and `playwright.config.ts`'s `webServer` entry (and
   whatever `package.json` script it calls) to learn EXACTLY how 1.5 serves the
   Lynx web output for tests today: which command builds it, which directory/file
   is served as the entry page, and how that host page references the bundle
   (grep across `e2e/` and `package.json` for `lynx-view`, `web-core`,
   `web-elements`, `.web.bundle`). This is ground truth - do not guess from the
   Lynx docs alone.
2. Using what you found in step 1, determine which shape this repo actually has:
   (a) `npm run build` already emits BOTH the bundle AND a ready-to-serve host
       `index.html` (+ its JS/CSS) into `dist/` - i.e. the rspeedy project's own
       `environments.web` config self-hosts. In this case the existing Dockerfile
       pattern (`COPY --from=web /app/dist ./pb_public`) needs at most a path
       rename, or
   (b) the host shell is a SEPARATE small build (its own output dir, e.g. built by
       whatever 1.5 set up under `e2e/`) that must run alongside `rspeedy build`
       and land its output next to the bundle before the final `COPY`.
   Whichever is true, make the Dockerfile's `web` stage run every build that is
   actually needed and land the FINAL combined static output in one directory
   that the last `COPY --from=web ... ./pb_public` line copies from. Reuse 1.5's
   already-proven mechanism; do not invent a third one.
3. The host page's `<lynx-view url="...">` MUST reference the bundle via a
   same-origin RELATIVE path (e.g. `/main.web.bundle`), never an absolute
   `http://localhost:PORT` URL - that pattern is fine for 1.5's local dev/test
   server but would hardcode a dead host into the production image. Grep the host
   HTML/TSX for the `url=` value passed to `<lynx-view>`. If it is an absolute dev
   URL baked into a shared template/config (trivially yours to parameterize to
   same-origin), fix it. If fixing it would require changing 1.5's E2E harness
   itself in a non-trivial way, STOP and report it as a blocking cross-unit
   finding instead of quietly forking the harness.
4. Rename the Dockerfile's `ARG VITE_PB_URL` / `ENV VITE_PB_URL` (currently lines
   ~14-15) to whatever env var unit 1.4 actually wired the PB client seam to read.
   Grep `src/lib/pb.ts` (or wherever 1.4 placed the seam) for the host-resolution
   variable name - it will NOT be `import.meta.env.VITE_*` since Vite is gone; it
   may be `process.env.PB_URL`, an rspeedy `source.define`-injected constant, or
   similar. Use that EXACT name; do not invent one. Update the Dockerfile comment
   (currently lines ~11-13) to match, keeping the same intent: empty = same-origin
   default, set only if the API lives on a different host than the web app.
   If 1.4 left PB host resolution entirely build-time-constant with NO override
   hook, do not add a new env-var mechanism yourself (that is feature work outside
   this unit's plumbing-only scope) - instead note this plainly as a finding for
   8.3 (the deploy doc must not reference a variable that no longer exists).
5. Base image: rspeedy requires Node 20.19+ or 22.12+ (verified 2026-07 against
   the `@lynx-js/rspeedy` changelog). `node:22-alpine` already satisfies this as
   long as the resolved Alpine tag tracks a current 22.x. If Docker is available
   (step-by-step in DONE MEANS), confirm with
   `docker run --rm node:22-alpine node -v`. Do not bump to `node:24-alpine`
   unless you find 22.x genuinely fails the floor - minimize the diff.
6. Confirm `docker-compose.yml`, `docker-compose.prod.yml`, and
   `.github/workflows/deploy.yml` need NO changes: none of them name `dist`,
   `vite`, or any Vite-specific path - they only reference `build: .` and the
   `dooey` service. Grep to confirm, then state explicitly in your CONTRACT note
