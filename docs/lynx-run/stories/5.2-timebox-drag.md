CURRENT (old) POINTER-EVENT IMPLEMENTATION -- study before porting:
- Day-view MOVE (TimeBlock, motion drag="y"): OFFSET-based. onDrag reads
  info.offset.y (delta from grab), previewFromOffset computes
  start = clamp(snap(item.startMin + offset.y/pxPerMin), DAY_START, DAY_END-SNAP)
  and off = (raw < DAY_START-20). onDragEnd: if off -> onUnschedule(); else if
  start changed -> onSchedule(start). NO getBoundingClientRect (pure delta).
- Day-view RESIZE (the hem): pointer events, OFFSET-based. onPointerDown captures
  pointer + records fromY + baseDur; onPointerMove computes
  dur = clamp(snap(baseDur + (clientY-fromY)/pxPerMin), SNAP, DAY_END-startMin);
  onPointerUp commits onResize(dur). NO rect.
- Shelf DROP (ShelfChip, motion drag): ABSOLUTE-position. onDragEnd -> onDropAt(
  info.point.x, info.point.y) -> minuteAtPoint uses gridRef.getBoundingClientRect()
  + window.scrollX/Y to convert page coords -> minute (returns null if outside).
- Week-view MOVE (WeekBlock, motion drag): ABSOLUTE-position. onDrag ->
  preview(dur, point.x, point.y); onDragEnd -> drop(task, fromDay, point.x, point.y)
  -> resolvePoint uses gridRef.getBoundingClientRect() + GUTTER + window.scroll to
  compute { day: days[floor(((x-left)/width)*7)], min } and patches start_min
  (and due_date if the day changed).
So: MOVE and RESIZE are delta-based (no measurement); SHELF-DROP and WEEK cross-day
need the grid's on-screen rectangle.

LYNX EQUIVALENTS (cited -- do NOT guess these APIs; crib: "when an API is uncertain
FETCH THE DOC PAGE"):
- Directive: a handler is a main-thread function when its first statement is the
  string 'main thread'. Ref: https://lynxjs.org/react/main-thread-script .
- Binding: main-thread:bindtouchstart / main-thread:bindtouchmove /
  main-thread:bindtouchend on the draggable `<view>`. Ref: same page + swiper
  example on https://lynxjs.org/guide/interaction/event-handling/manipulating-element.react.html
- Touch coords on the event object: event.touches[0] exposes clientX/clientY
  (window coords), pageX/pageY (LynxView coords) and x/y (element-relative). Ref:
  https://lynxjs.org/api/lynx-api/event/touch-event .
- Element handle: event.currentTarget and event.target are MainThread.Element.
  Style writes: event.currentTarget.setStyleProperty('transform',
  `translate(0px, ${dy}px)`) (kebab property, string value). Refs: MTS page;
  https://lynxjs.org/api/lynx-api/main-thread/main-thread-element .
- Refs: useMainThreadRef<MainThread.Element>() + main-thread:ref={ref} to hold the
  grid handle and per-gesture state. Access ref.current only inside main-thread
  functions. Ref: MTS page.
- ELEMENT MEASUREMENT (the getBoundingClientRect replacement): there is NO sync
  getBoundingClientRect on MainThread.Element. Measure via
  ref.current.invoke({ method: 'boundingClientRect',
     params: { relativeTo: 'screen' },
     success: (res) => { /* res = { left, top, width, height, right, bottom } */ },
     fail: (err) => { /* log */ } }).exec();
  It is CALLBACK-based (async), even on the main thread. Ref:
  https://lynxjs.org/api/lynx-api/main-thread/main-thread-element (invoke) and
  https://lynxjs.org/guide/interaction/event-handling/manipulating-element.react.html
  (the boundingClientRect example). Because it is async, MEASURE ONCE at
  touchstart and cache the rect in a useMainThreadRef; reuse the cached rect
  synchronously during touchmove/touchend. Because clientX/clientY are window
  coords and boundingClientRect(relativeTo:'screen') is screen-relative, the old
  window.scrollX/scrollY math is DROPPED -- subtract rect.left/top directly.
- Commit to the background thread (React state / PB write): call
  runOnBackground(fn)(args) from inside the main-thread touchend handler; fn runs
  on the background thread and there calls useUpdateTask().mutate(...). Ref:
  MTS page (runOnBackground). Captured vars must be JSON-serializable; main-thread
  functions run only after TTI, no nested defs, cannot mutate outer-scope vars
  (crib MTS constraints) -- so hold mutable gesture state in useMainThreadRef.current
  fields, not closure variables.

SPEC (numbered; concrete; cite MTS constraints):
1. GESTURE STATE lives in a useMainThreadRef (per draggable) holding a plain
   JSON-serializable object: { startY, startX, gridLeft, gridTop, gridWidth,
   gridHeight, baseStartMin, baseDur, moved }. Initialize/measure at touchstart.
   Never capture-and-mutate a closure variable inside a main-thread fn (crib
   constraint "cannot mutate outer-scope vars").
2. DAY-VIEW MOVE (TimeBlock). Attach to the block `<view>`:
   - main-thread:bindtouchstart(e): 'main thread'; store startY = e.touches[0].clientY,
     baseStartMin = <the block's start_min, passed in as a JSON-serializable prop>,
     moved=false. Lift look: e.currentTarget.setStyleProperty('z-index','30');
     setStyleProperty('box-shadow', '<the whileDrag shadow>').
   - main-thread:bindtouchmove(e): 'main thread'; dy = e.touches[0].clientY - startY;
     if |dy|>4 moved=true; e.currentTarget.setStyleProperty('transform',
     `translateY(${dy}px)`). Compute the previewed slot
     start = clamp(snap(baseStartMin + dy/pxPerMin), DAY_START, DAY_END-SNAP),
     off = (baseStartMin + dy/pxPerMin) < DAY_START-20, and drive the ghost `<view>`
     via its own MainThreadRef.setStyleProperties({ top, height, opacity }) (pxPerMin
     is a number -> pass it in as a serializable prop, do not read React context on
     the main thread).
   - main-thread:bindtouchend(e): 'main thread'; reset transform/z-index/shadow to
     ''; hide ghost; then commit on the background thread:
     runOnBackground((payload) => { if (payload.off) unschedule(id); else if
     (payload.start !== payload.baseStartMin) schedule(id, payload.start); })(
     { off, start, baseStartMin }). schedule/unschedule are the api.ts mutations
     (useUpdateTask patch { start_min } / { start_min: 0 }). If moved is false,
     do NOT commit -- let the block's bindtap navigation run (SPEC 6).
3. DAY-VIEW RESIZE (the hem `<view>`, separate from the block body). Same three
   main-thread handlers, delta on clientY, baseDur from prop:
   dur = clamp(snap(baseDur + (e.touches[0].clientY-startY)/pxPerMin), SNAP,
   DAY_END - baseStartMin). touchmove drives the block height + ghost; touchend
   runOnBackground -> resize(id, dur) (useUpdateTask patch { dur_min: dur }). Stop
   propagation so the hem gesture never triggers the block-move gesture (the hem is
   a child `<view>`; put its own bind handlers on it -- main-thread events bubble,
   so early-return the parent handler when a resize is active via a shared
   MainThreadRef flag, mirroring the old e.stopPropagation()).
4. DRAG-OFF-TO-UNSCHEDULE is folded into SPEC 2 (off = dragged above DAY_START-20).
   On off, also show the ArrowUpFromLine affordance already in the static markup
   (toggle opacity/saturate via setStyleProperty during touchmove).
5. SHELF-DROP (ShelfChip) and WEEK cross-day (WeekBlock) -- these need the grid
   rectangle:
   - touchstart: measure the grid ONCE. The grid `<view>` (HourRules container) is
     held in a useMainThreadRef `gridRef` (add main-thread:ref to it in the render).
     Call gridRef.current.invoke({ method:'boundingClientRect',
       params:{ relativeTo:'screen' },
       success:(r)=>{ store gridLeft=r.left, gridTop=r.top, gridWidth=r.width,
                       gridHeight=r.height } }).exec();
     Because invoke is async, START the drag optimistically and use the cached rect
     from the previous frame if the first success has not landed; the rect does not
     change mid-gesture (no scroll during a drag) so a one-frame delay is
     acceptable. (If measuring at touchstart proves too late in practice, measure at
     mount instead and refresh on scroll -- note which you chose.)
   - touchmove/touchend: convert window coords to grid-relative:
     localX = e.touches[0].clientX - gridLeft; localY = e.touches[0].clientY - gridTop.
     Shelf-drop minute (day sheet, single column):
       if localX<0 || localX>gridWidth || localY< -10 || localY>gridHeight+10 ->
       out of grid, cancel; else min = clamp(snap(DAY_START + localY/pxPerMin),
       DAY_START, DAY_END-SNAP) and runOnBackground -> b.onSchedule(min).
     Week cross-day (7 columns over GUTTER margin):
       usableLeft = gridLeft + GUTTER; usableWidth = gridWidth - GUTTER;
       col = clamp(floor(((e.touches[0].clientX - usableLeft)/usableWidth)*7),0,6);
       min = clamp(snap(DAY_START + localY/pxPerMin), DAY_START, DAY_END-SNAP);
       runOnBackground -> mutate({ id, patch:{ start_min:min, ...(days[col]!==fromDay
       ? { due_date: toPbDate(days[col]) } : {}) } }). days[]/fromDay/GUTTER passed
       as serializable props/values.
6. TAP vs DRAG: the block body wraps a <Link> (navigate to /task/$id). Guard it: in
   touchend, if moved===true, suppress navigation (set a MainThreadRef `justDragged`
   and have the Link's bindtap early-return one tick), replacing the old
   onClickCapture/dragging trick. If suppression across the MTS/background boundary
   is unreliable on the web target, STOP-on-ambiguity rather than shipping a block
   that navigates on every drop.
7. TEXT/STYLE: any `<text>` added (e.g. the ghost's time label) sets explicit
   color/size (crib: text does not inherit). Keep durations/easings within
   CLAUDE.md motion discipline (interactions <= 200ms).
8. DEAD CODE: remove motion/pointer imports now unused; no commented code; no TODO.
