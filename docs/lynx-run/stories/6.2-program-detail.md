SPEC:

1. General element-mapping rules for every file in this unit (apply uniformly, do not
   re-derive per file):
   - Every `<div>`/`<section>`/`<span>` that is purely structural -> Lynx `<view>`.
   - Every text-bearing node (`<p>`, `<span>` with text, `<h1>`, `<li>` text) -> Lynx
     `<text>` with its OWN explicit `color`/`font-family`/`font-size` (text does not
     inherit CSS from ancestors - crib sheet, verified against
     https://lynxjs.org/api/elements/built-in/text.html). A nested `<text>` INSIDE a
     `<text>` DOES inherit `color`/`font-family` from its parent `<text>` and can
     override them - use this for any inline run of styled text within one paragraph
     (see SPEC 6 for Markdown's bold/code spans).
   - `line-clamp-*` Tailwind utilities -> the Lynx `<text text-maxline={N}>`
     attribute, not a CSS class (same as unit 6.1 SPEC 3).
   - No `hover:`/`group-hover:`/`focus-visible:` Tailwind variants - Lynx has no
     pseudo/data-attr selectors (crib; https://lynxjs.org/rspeedy/styling says the
     preset strips unsupported utilities at build time, so these would silently no-op
     even if left in). Where the old code used hover for a reveal, show the element
     always visible instead (same call as unit 6.1 SPEC 3's paintbrush button).
   - `motion/react` (`motion.div`, `AnimatePresence`, `whileTap`, spring transitions)
     -> CSS transitions/keyframes for anything declarative (enter/exit/settle -
     PLAN 5.3). None of this unit's animations are gesture-linked, so none of them
     need Main-Thread Scripting.
   - For any discrete press/tap depress affordance, reuse the exact mechanism
     unit 2.3/4.2 already established (same instruction as 6.1 SPEC 3) - do not invent
     a new one.

6. Port Markdown.tsx - READ THIS BEFORE WRITING CODE. src-legacy's Markdown.tsx (94
   lines) supports exactly these constructs; port each to the stated Lynx mapping,
   and nothing else (no tables, images, links-as-links, code blocks, nested lists,
   or horizontal rules are supported today - do not add them):

   | Construct (src-legacy regex/logic)          | Lynx mapping                                                                 |
   |----------------------------------------------|-------------------------------------------------------------------------------|
   | `#`/`##` heading (`level <= 2`)               | `<view>` > `<text>` styled like the old `<h4>` (font-display, text-xl, semibold, tracking-tight, color ink) - explicit styles, no semantic heading tag exists on Lynx. |
   | `###`-`######` heading (`level > 2`)          | `<view>` > `<text>` styled like the old `<h5>` (10px, uppercase, tracking-[0.22em], ink-muted). |
   | Bullet list item (`-`/`*` prefix)              | A `<view>` per list (flex-col, vertical gap) containing one `<view>` (flex-row) per item: a leading `<text class="text-leaf">-</text>` (the literal em-dash marker, replacing the old `before:content-['-']` pseudo-element - no pseudo-elements on Lynx) + the item's `<text>` content. Do NOT use Lynx's `<list>` element - it is a virtualized component, the wrong tool for a handful of markdown bullets. |
   | Numbered list item (`\d+\.` prefix)            | Same `<view>` stack shape as bullets, but the leading marker `<text>` is the literal number + "." (there is no CSS `list-style: decimal` equivalent - compute and render the number yourself, `i + 1`). |
   | Blockquote (`>` prefix)                        | `<view>` with a left-border style (a 2px-wide colored `<view>` sibling, since Lynx has no `border-left` shorthand guarantee across all targets - use an explicit absolutely-positioned or flex-adjacent bar `<view>` if a plain `border-left` CSS property does not render correctly; verify visually) containing the quoted `<text>` (italic, ink-muted). |
   | Plain paragraph line                           | `<view>` > `<text>` (15px, leading-relaxed, ink-muted). |
   | Blank line                                     | Flushes any open list (unchanged logic - this is pure parsing, no Lynx concern). |
   | Inline `**bold**`                              | A NESTED `<text>` inside the paragraph's parent `<text>`, with `font-weight: 600`/`color: ink` overriding the parent - relies on the confirmed nested-`<text>` inheritance/override behavior (fact in unit 6.1's CONTEXT). |
   | Inline `` `code` ``                            | A nested `<text>` with `font-family` set to the ported monospace font token and `color: ink`. The old look also had a `bg-ink/5` chip background on the `<code>` span - background-color on an INLINE nested `<text>` is unconfirmed territory (the fetched Lynx `<text>` docs did not address it). Try it; if the background does not render on the nested span, DROP just the background (keep the monospace font + color change) - the font-family switch alone still reads as "code". Note whichever way it landed in this unit's result, do not spend more than one attempt on it. |
   | Bare/plain text passthrough (`Fragment`)       | Just the text content inside the current `<text>` node - no extra wrapper. |

   Parsing logic itself (`renderBlocks`, `inline`, the `flushList` state machine, all
   the regexes) is pure string/array logic - port UNCHANGED; only the JSX return
   values change shape per the table above.
