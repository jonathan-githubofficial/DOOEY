SPEC (concrete):

1) `src/features/auth/api.ts` - PORT `src-legacy/features/auth/api.ts` verbatim; it is
   framework-agnostic and the session contract is AUTHORITATIVE (CLAUDE.md
   "Auth & sessions"). Preserve EXACTLY:
   - `initSession()`: if `!pb.authStore.isValid` return; else `await
     pb.collection("users").authRefresh()`; in `catch`, drop the session ONLY when
     `e instanceof ClientResponseError && e.status >= 400 && e.status < 500`
     (`pb.authStore.clear()`). A network failure or a 5xx MUST NOT sign the user out.
     Do not add extra catch branches, retries, or guards (CLAUDE.md "Fix the root
     cause / Trust the boundaries"). If the Lynx fetch seam (1.4) surfaces network
     errors as something OTHER than a non-`ClientResponseError` throw (i.e. if a
     dropped connection could arrive as a `ClientResponseError` with a 4xx-looking
     status), STOP (status "blocked") - a wrong classification here silently signs
     people out. Otherwise port as-is.
   - `signIn(email, password)`: `pb.collection("users").authWithPassword(...)`.
   - `signUp(email, password)`: `create({ email, password, passwordConfirm: password })`
     then `signIn(...)`.
   - `signOut()`: `pb.authStore.clear()`.

2) `initSession()` on boot - EDIT the app entry (the file that renders
   `<RouterProvider/>`, from 3.1). Add `void initSession();` BEFORE `root.render(...)`,
   mirroring `src-legacy/main.tsx` line 17. Because `pb.authStore` is already
   rehydrated from the storage adapter (1.4) at import time, the guard in 3.1 sees a
   valid session synchronously on boot; `initSession()` then refreshes it in the
   background and, via `pb.authStore.onChange -> router.invalidate()` (wired in 3.1),
   re-runs the guard if a definitive 4xx clears it. Do not `await` it before render
   (offline/flaky boot must still paint). If the storage adapter is ASYNC and
   `pb.authStore.isValid` is therefore `false` at first synchronous read (so a
   returning user briefly hits `/login`), STOP and report it as a finding - the fix
   is a hydration gate that belongs to the 1.4 seam, not an improvisation here.

3) `SignInCard` on Lynx (crib "Elements"; PORT
   `src-legacy/features/auth/components/SignInCard.tsx`). Keep the state machine
   exactly: `mode` in|up, `email`, `password`, `busy`, `error`; `submit()` guards
   `!email.trim() || !password || busy`, sets busy, calls `signUp`/`signIn` then
   `await syncWithPB()` then `onSignedIn()`, and on throw sets `error` +
   `busy=false`. Element mapping:
   - `Panel`, `Eyebrow`, `Button` from L2 primitives; `Input` from L2 (`Input` was
     rebuilt on the Lynx `<input>` in 2.3).
   - Text nodes (`<h2>`, mode-toggle button label, error `<p>`) -> `<text>` with
     EXPLICIT color/font-family/size (crib: text does not inherit). Error text uses
     `text-clay`.
   - Inputs: the Lynx `<input>` reports value via `bindinput` (event.detail.value),
     NOT React `onChange`/`e.target.value`. The L2 `Input` primitive already adapts
     this - use ITS documented prop (a `value` + a change callback). Follow the 2.3
     `Input` API; do not hand-roll a second input adapter. Password masking:
     `<input>` `type="password"` on the Lynx web target; if the pinned Lynx `<input>`
     lacks a password type, use its `password` boolean prop per the 2.3 Input - if
     neither exists, STOP (do not ship a plaintext password field).
   - Submit trigger: `Button` press via the primitive's tap binding (`bindtap` under
     the hood). Enter-to-submit (old `onKeyDown` Enter) is a nicety - wire it via the
     `<input>` confirm event if the L2 Input exposes it; otherwise DROP it (record in
     broom) - the button is the primary path.
   - Autocomplete attributes (`autoComplete="email"` etc.) are DOM-only; drop them on
     Lynx (record). No behaviour depends on them.

4) `src/pages/Login.tsx` - COMPLETE the shell from 3.1:
   - `const router = useRouter(); const { redirect } = route.useSearch();`
     (`route = getRouteApi("/login")`, already in the shell).
   - Render the centered wordmark (kept from 3.1) + `<SignInCard onSignedIn={() =>
     router.history.push(redirect ?? "/")} />`. This is the round-trip: `redirect` is
     a full href string (path+search) captured by the guard (3.1), so push it through
     `router.history` (NOT a typed `navigate({ to })`), matching the src-legacy
     comment. On memory history `router.history.push("/calendar")` returns the user to
     the intended space. If `router.history.push` is unavailable on the memory-history
     router at the pinned version, STOP (this is the load-bearing return path).

5) Session survives reload (CLAUDE.md "Auth & sessions"). No new code - this is the
   consequence of: (a) `pb.authStore` persisted via the 1.4 storage adapter, (b)
   `initSession()` refresh on boot, (c) the guard passing when `isValid`. NOTE for the
   E2E author: on memory history a reload restarts at `initialEntries[0]` ("/"), so a
   reloaded, still-authenticated user lands on the guarded index (Today's interim from
   3.1 / real Today from 4.1) - NOT on their previous route, and NOT on `/login`.
   Assert "still signed in", not "same route".
