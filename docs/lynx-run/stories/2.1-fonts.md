# 2.1 - Fonts (static per-weight instances)

LAYER: L2-design
TIER: normal
RISK: low-medium. Mechanical CSS/asset work; the only real risk (native font rendering) is
out of mechanical reach on this Windows box and is PARKED.

GOAL: Replace `@fontsource-variable/{outfit,fraunces}` with static per-weight font files,
registered via hand-written `@font-face` rules (one family name per weight, since Lynx
ignores `font-weight`), and wire a small CSS layer so existing Tailwind class usage
(`font-sans`, `font-display`, `font-normal`/`font-medium`/`font-semibold`/`font-bold`/
`font-black`) keeps resolving to the correct family without every future unit having to
rename classes.

CONTEXT: L1 is complete. `src-legacy/` is the read-only old app (unit 1.1 did `git mv src
src-legacy`). In the new `src/`, unit 1.1 already: scaffolded ReactLynx via rspeedy, wired
Tailwind v3 + `@lynx-js/tailwind-preset`, and ported the color design tokens (`--paper`,
`--ink`, `--leaf`, `--zest`, `--radius-card`, `.shadow-soft`, etc.) from the old v4
`@theme inline` block into `tailwind.config.js` + `src/styles/global.css` (plain CSS custom
properties, which are portable as-is). This unit only adds fonts - it does not touch color
tokens. No router exists in the new tree yet (unit 3.1); this unit has no screen of its own -
its output is verified visually by unit 2.3's Gallery (see DONE MEANS).

IN SCOPE (files):
- Old (read-only reference): `src-legacy/styles/global.css` (the `--app-font-sans` /
  `--app-font-display` vars and the `@theme inline` font mapping), `src-legacy/main.tsx`
  (the `@fontsource-variable/*` imports), `package.json` (current font deps).
- New (create/edit):
  - `src/assets/fonts/` (new dir) - the 6 static font files.
  - `src/styles/fonts.css` (new) - `@font-face` rules + the class-mapping rules.
  - `src/styles/global.css` (edit) - add `@import "./fonts.css";`, update
    `--app-font-sans` / `--app-font-display` default values.
  - `package.json` (edit) - remove `@fontsource-variable/{outfit,fraunces}`, add
    `@fontsource/{outfit,fraunces}` (used only as a source of raw font files, see SPEC).
  - Entry file created by 1.1 (`src/main.tsx` or `src/index.tsx` - whichever rspeedy
    scaffolded; check with `ls src/*.tsx`) - remove any leftover `@fontsource-variable`

OUT OF BOUNDS:
- Adding explicit `font-sans`/`font-display` classes to any page/feature `<text>` (that is
  each later unit's own job when it ports that component - Lynx text does not inherit CSS,
  so this is a permanent cross-cutting reminder, not a one-time fix here).
- `src/features/style/tokens.ts` `FONT_STACKS` (still says `"Outfit Variable", "Outfit"` /
  `"Fraunces Variable", "Fraunces"` - that array only feeds the Style Studio font picker,
  which is unit 3.4's screen. Leave it; 3.4 must update the stack strings to the new
  per-weight family names when it ports StyleStudio.
- `tailwind.config.js` `theme.extend.fontFamily`/`fontWeight` keys - not touched (see SPEC
  step 6 for why the plain-CSS route was chosen over a Tailwind plugin).

BROOM (old file -> disposition):
- `@fontsource-variable/outfit`, `@fontsource-variable/fraunces` (package.json deps): DROP.
  Variable-axis weight switching relies on the browser interpolating `font-weight` inside one
  variable font file; Lynx's `@font-face` explicitly "does not support font-style, font-weight
  and font-variant" (https://lynxjs.org/api/css/at-rule/font-face.html), so a variable font's
  weight axis is permanently inert there. PLAN 5.7.
- `src-legacy/main.tsx`'s two `import "@fontsource-variable/..."` lines: DROP. Font loading on
  Lynx is CSS-only (`@font-face` in `fonts.css`); there is no JS-side font import.
- `@fontsource/outfit` and `@fontsource/fraunces`'s OWN generated per-weight CSS (e.g.
  `@fontsource/outfit/600.css`): DROP, do not import these files. Fontsource's static packages
  still emit one shared `font-family: "Outfit"` per weight file, distinguished only by
  `font-weight` in the `@font-face` block - the exact pattern Lynx ignores. Only the raw
  `.woff2` files from the package are used; this unit hand-writes the `@font-face` rules with
  distinct family names (`"Outfit-400"`, `"Outfit-600"`, ...) per the crib sheet's example.
- Old `global.css`'s `@theme inline { --font-sans: var(--app-font-sans); --font-display: ... }`
  Tailwind v4 mapping: already superseded by 1.1's v3 port - nothing left to do here.

SPEC:

1. Confirm the new tree's entry file: run `ls src/*.tsx` in the worktree root. rspeedy
   scaffolds typically use `src/App.tsx` as the root component; the actual bootstrap/import
   file may be `src/index.tsx` or inlined in `src/App.tsx` - use whichever file currently holds
   top-level imports. This is a one-line lookup, not a judgment call.

2. Install static font packages and remove the variable ones:
   ```
   npm install @fontsource/outfit @fontsource/fraunces
   npm uninstall @fontsource-variable/outfit @fontsource-variable/fraunces
   ```

3. Locate the raw weight files fontsource shipped:
   ```
   ls node_modules/@fontsource/outfit/files/
   ls node_modules/@fontsource/fraunces/files/
   ```
   Fontsource v5 static packages name files `<id>-<subset>-<weight>-<style>.<ext>`, e.g.
   `outfit-latin-400-normal.woff2`. Fraunces is a variable family with an optical-size (opsz)
   axis; its static package may expose more than one file per weight (different opsz cuts). If
   you see multiple candidates for a weight, pick the one whose name has no extra axis suffix
   (the "standard"/display-optical-size default cut) and record which exact filename you chose
   in the unit's commit message - this is the only place in this story where a small pick is
   left to the executor, and it is low-risk (visual-only, easily swapped later).

4. Copy exactly 6 files into a new `src/assets/fonts/` directory, renamed for clarity:
   - `outfit-latin-400-normal.woff2` -> `src/assets/fonts/outfit-400.woff2`
   - `outfit-latin-500-normal.woff2` -> `src/assets/fonts/outfit-500.woff2`
   - `outfit-latin-600-normal.woff2` -> `src/assets/fonts/outfit-600.woff2`
   - `outfit-latin-700-normal.woff2` -> `src/assets/fonts/outfit-700.woff2`
   - Fraunces 700 (bold) file -> `src/assets/fonts/fraunces-700.woff2`
   - Fraunces 900 (black) file -> `src/assets/fonts/fraunces-900.woff2`
   (Only woff2 is needed for this unit - the web target is the only one verified on this
   machine. Native TTF/OTF sourcing is PARKED, see below.)

5. Create `src/styles/fonts.css`:
   ```css
   /* One @font-face per weight instance - Lynx ignores font-weight/font-style/font-variant
      on @font-face, so each weight needs its own family name.
      https://lynxjs.org/api/css/at-rule/font-face.html */
   @font-face {
     font-family: "Outfit-400";
     src: url("../assets/fonts/outfit-400.woff2") format("woff2");
   }
   @font-face {
     font-family: "Outfit-500";
     src: url("../assets/fonts/outfit-500.woff2") format("woff2");
   }
   @font-face {
     font-family: "Outfit-600";
     src: url("../assets/fonts/outfit-600.woff2") format("woff2");
   }
   @font-face {
     font-family: "Outfit-700";
     src: url("../assets/fonts/outfit-700.woff2") format("woff2");
   }
   @font-face {
     font-family: "Fraunces-700";
     src: url("../assets/fonts/fraunces-700.woff2") format("woff2");
   }
   @font-face {
     font-family: "Fraunces-900";
     src: url("../assets/fonts/fraunces-900.woff2") format("woff2");
   }

   /* Class-based weight -> family mapping, so existing className usage
      (font-sans/font-display combined with font-normal/medium/semibold/bold/black)
      keeps resolving without every consumer renaming classes. Compound class selectors
      (.a.b) are plain CSS, not pseudo/data-attr selectors, so they are Lynx-legal:
      https://lynxjs.org/api/css/selectors.html only restricts pseudo-CLASSES
      (:active/:not()/:root supported, others not) and there is no pseudo-ELEMENT support
      at all - neither restriction applies to compound class selectors. */
   .font-sans,
   .font-sans.font-normal {
     font-family: "Outfit-400", system-ui, sans-serif;
   }
   .font-sans.font-medium {
     font-family: "Outfit-500", system-ui, sans-serif;
   }
   .font-sans.font-semibold {
     font-family: "Outfit-600", system-ui, sans-serif;
   }
   .font-sans.font-bold {
     font-family: "Outfit-700", system-ui, sans-serif;
   }
   /* Fraunces ships bold/black only (PLAN 5.7) - no regular instance. CLAUDE.md's design
      aesthetic section already only ever describes display type as "bold/black", so a bare
      .font-display (no weight class) defaults to bold. */
   .font-display,
   .font-display.font-bold {
     font-family: "Fraunces-700", Georgia, serif;
   }
   .font-display.font-black {
     font-family: "Fraunces-900", Georgia, serif;
   }
   ```

6. Why plain CSS instead of a Tailwind plugin: Tailwind's `fontWeight` utilities
   (`font-normal`/`font-medium`/.../`font-black`) still compile to a `font-weight: N`
   declaration - harmless on Lynx text (the property is simply ignored, per the font-face doc),
   costs nothing, and does not require touching the Tailwind config or its plugin API (which
   `@lynx-js/tailwind-preset`'s own docs describe as "still not stable, use with caution").
   The rules in step 5, appended after `@import "tailwindcss"` via `global.css`, are ordinary
   compound-class CSS with higher specificity than Tailwind's own single-class utilities, so
   they always win regardless of import order. Net effect: `className="font-display
   font-black"` (already used throughout the old app, e.g. `dock.tsx`, `masthead.tsx`) needs
   zero changes when later units port those files.

7. Edit `src/styles/global.css`:
   - Add `@import "./fonts.css";` directly under the existing `@import "tailwindcss";` line.
   - In the `:root` block, update (or add, if 1.1 did not carry these two vars over) the
     default font vars:
     ```css
     --app-font-sans: "Outfit-400", system-ui, sans-serif;
     --app-font-display: "Fraunces-700", Georgia, serif;
     ```
     These stay as the CSS-variable indirection layer the old app used (`--font-sans`,
     `--font-display` in the Tailwind theme resolve to these) - keep the indirection so a
     future Style Studio "custom font" override (unit 3.4) still has one place to patch.

8. Remove the old imports found in step 1's entry file (`import
   "@fontsource-variable/outfit"; import "@fontsource-variable/fraunces";` or equivalent) -
   there is nothing to replace them with; font loading is now purely declarative CSS via
   `fonts.css`.
