SPEC (concrete):

1) Read-only `DoodleSvg` (`src/components/doodle-svg.tsx`) - PORT
   `src-legacy/components/doodle-svg.tsx` onto the Lynx `<svg>` element (v3.7; crib
   "SVG"; https://lynxjs.org/next/blog/lynx-3-7 - FETCH if the `<svg>`/`<path>` prop
   surface is uncertain). Keep: `viewBox="0 0 100 100"`, `preserveAspectRatio="none"`,
   one `<path>` per stroke with `d={strokePath(s.points)}`, `stroke="hsl(var(--<color>))"`,
   `strokeWidth`, `strokeLinecap/Linejoin="round"`, `opacity={0.85}`. The
   `vectorEffect="non-scaling-stroke"` (`relative` false) branch: if Lynx `<svg>` does
   not support `vectorEffect`, keep only the `relative` (viewBox-unit width) path used
   by dock/avatar and DROP the non-scaling branch here (record; the task-page usage is
   4.x). No gestures, no MTS - static paths only. If Lynx `<svg>` cannot render a
   basic `<path d>` at the pinned engine version, STOP and report (this blocks
   avatar/page-doodle display).

2) `Masthead` (`src/components/masthead.tsx`) - PORT

3) `Dock` (`src/components/dock.tsx`) - PORT `src-legacy/components/dock.tsx`.

     four `DockTab`s from the `spaces` array (Planner "/", Calendar "/calendar",
     Boards "/boards", Projects "/projects").

     `bottom-[max(1rem,env(safe-area-inset-bottom))]`. `env(safe-area-inset-*)` does
     NOT exist on Lynx (CLAUDE.md native section; crib). Use the Lynx safe-area inset
     mechanism established in 2.3 (safe-area) for the bottom offset. If 2.3 exposed a
     token/util, use it; if not, STOP (do not hardcode 0 - the dock would sit under
     the home indicator on device; but note verification is web-only, so on web the
     inset is 0 and that is expected).

     checks mapping "/projects","/calendar","/board*","/account"|"/style" -> parent

   - `<Link>` -> TanStack `Link` (works on the memory router); `aria-*`/`title` are
     DOM-only - drop or map to Lynx accessibility props if 2.x defined any (record).

4) Dock motion (crib "Animation": CSS transitions for non-gesture, MTS for
   gesture-linked; PLAN 5.3):
   - The gliding active highlight (old `motion.span layoutId="dock-active"` spring)
     is ROUTE-driven, not gesture-driven -> re-author as a single absolutely-
     positioned highlight `<view>` whose `transform: translateX(...)` + width
     transition to the active stop via a CSS transition (small overshoot easing per
     CLAUDE.md "settle, don't snap"). Compute the active stop's offset/width from its
     index. Do NOT use `layoutId` (no motion lib).
   - The active-tab label reveal (old `AnimatePresence` width 0->auto) -> CSS
     transition on max-width + opacity (auto-width is not animatable; use a max-width
     large enough for the longest label). Non-gesture.
   - Press-depress on each tab / the cluster (old `active:scale-95`) is GESTURE-linked
     -> MTS (crib "Animation": `'main thread'` directive, `main-thread:bindtouchstart`/
     `bindtouchend`, `event.currentTarget.setStyleProperty("transform","scale(0.95)")`
     / reset; https://lynxjs.org/react/main-thread-script - FETCH for exact event
     names). MTS constraints: captured vars JSON-serializable, no outer-scope
     mutation, runs after TTI. Keep it tiny (scale only). If MTS press proves flaky on
     the web target, a CSS transition on a pressed class via the preset's `uiVariants`
     is an acceptable fallback (record which).
