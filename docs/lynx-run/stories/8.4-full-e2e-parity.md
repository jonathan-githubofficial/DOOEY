# 8.4 - Full E2E parity gate

LAYER / TIER / RISK: L8-cutover / normal (sonnet) / MEDIUM-HIGH (this is the
run's functional closing gate - it both writes a judgment-heavy checklist AND is
allowed a bounded amount of real bug-fixing to reach parity, which is more
authority than most L8 units get).

GOAL: Produce `docs/lynx-run/parity-checklist.md` mapping every item in
CLAUDE.md's "Current scope (locked feature list)" and every row of PLAN section
2's before/after table to an E2E spec (or an explicit, justified gap ruling);
fill genuinely thin coverage with new `@l8`-tagged specs; get the FULL Playwright
suite green from a fresh disposable PocketBase data dir.

CONTEXT (what exists when this unit starts):
- L1-L7 are committed and E2E-green through @l7, each having added `@lN`-tagged
  specs per the harness's cumulative-tag gating. Units 8.1 (Docker/deploy), 8.2
  (final broom - `src-legacy/` is gone, dead code swept), and 8.3 (CLAUDE.md +
  docs updated) already ran earlier in this same layer's unit order - so by the
  time this unit starts, CLAUDE.md's "Current scope" list reflects the
  post-migration wording 8.3 wrote, and there is no more `src-legacy/` to
  consult.
- CLAUDE.md's "Current scope (locked feature list)" names 7 items, but that list
  is the TARGET scope, not a guarantee every item was actually built
  pre-migration. `docs/roadmap.md` is ground truth for delivery status (as of
  its last entry before this plan was written: R1 navigation shell/tasks and R2
  task pages were delivered; R3 recurring habits was "in progress next"; R4
  journal, R5 Google Calendar two-way sync, and R6 in-app learning generation
  were not yet built). RE-VERIFY this against the actual current
  `docs/roadmap.md` when you run - do not trust this story's snapshot, things may
  have shipped since this story was written.
- "Full feature parity on the web target" (the orchestration framework's
  decisions record, item 2) means: whatever `src-legacy` actually implemented
  before this migration must work identically on Lynx. It does NOT mean "build
  whatever CLAUDE.md's aspirational locked-scope list describes but that never
  actually existed" - that would be new product development smuggled into a
  migration unit, which is out of bounds (see OUT OF BOUNDS).

IN SCOPE (files):
- `docs/lynx-run/parity-checklist.md` - NEW file, this unit's primary written
  deliverable.
- `e2e/*.spec.ts` - add new specs tagged `@l8` for any Section A/B row (see SPEC)
  with no real coverage; extend existing specs only if the extension is a small,
  additive assertion (do not rewrite earlier layers' specs wholesale - if an
  earlier layer's spec is actually wrong, that is a finding for its own layer,
  not a rewrite you do here).
- `src/` - small, obviously in-scope fixes ONLY when needed to make a
  locked-scope feature actually match `src-legacy` behavior (see OUT OF BOUNDS
  for the line you must not cross).
- `CLAUDE.md`, `docs/lynx-migration.md` section 2 - READ ONLY inputs.

OUT OF BOUNDS:
- Building NEW product features that never existed in `src-legacy` (e.g. actual
  Google Calendar OAuth + sync, an actual food journal, actual in-app learning
  generation) just because CLAUDE.md's locked-scope list names them. If
  `docs/roadmap.md` says a feature was not yet built pre-migration, it is a
  **GAP - pre-existing** ruling in the checklist, not a build task. This is the
  single most important scope boundary in this unit - violating it means doing
  unplanned, unreviewed product work inside a migration run.
- Any fix that touches auth/session semantics, PocketBase rules, `pb_migrations`,
  or `pb_hooks`, or that changes a data shape - per the orchestration framework's
  hard rules, this is a STOP/blocked situation, not something to patch through.
  Rule the checklist row BLOCKED with the exact question instead.
- `Dockerfile` / compose / deploy.yml (8.1), the broom (8.2, already done),
  CLAUDE.md prose beyond what 8.3 already wrote (do not re-edit 8.3's work here),
  the native runbook (8.5).
- `pb/pb_data` - never touch; the disposable PB dir is the only mutable
  environment (see DONE MEANS).

BROOM: N/A directly, but the checklist's gap-rulings function like a broom
decision for FEATURES rather than files - cite `docs/roadmap.md` for every
"GAP - pre-existing" ruling, the same way `docs/lynx-migration.md` section 4
requires a recorded reason for every dropped file.

SPEC (numbered, concrete):
1. Build the checklist skeleton in `docs/lynx-run/parity-checklist.md`, two
   sections, each a markdown table with columns
   `| Item | Status | Spec(s) | Note |`:
   - **Section A - locked feature list** (from CLAUDE.md's "Current scope"):
     Navigation shell, Tasks (Planner space), Task pages, Recurring habits, Food
     journal (Journal space), Google Calendar two-way sync, Learning generation
     in-app. For each, first check `docs/roadmap.md`'s delivery status (grep for
     its R-number and any "Delivered"/"in progress"/checkmark marker). If it was
     NOT actually built pre-migration, rule it:
     `Status = GAP - pre-existing (see roadmap.md R<n>)`, `Note` = one line
     citing why, and move on - do not write a spec for it, do not build it.
     If it WAS built pre-migration, find the E2E spec(s) that exercise the REAL
     acceptance criteria roadmap.md states for it (not just "the page renders") -
     grep `e2e/` for the feature name and its layer tag.
   - **Section B - PLAN section 2 table rows**: Build tool, Framework, Native
     shell (rule this one `PARKED - see native-hosts-runbook.md`, it is unit
     8.5's territory, never web-testable), Web target, Routing, Client state,
     Server state, Data SDK, Realtime, Markup, Styling, Animation,
     Gestures/drag, Doodle canvas, Paper grain, UI primitives, Icons, Fonts,
     Persistence, Backend (rule this one `N/A - unchanged`, PocketBase was never
     touched). For every other row, name the spec(s) that actually exercise that
     tech swap on the web target today.
2. For any row with NO real coverage, or only a smoke-level check (e.g. a page
   renders but the roadmap's actual "Acceptance" bullet is never asserted),
   write new Playwright specs tagged `@l8` in `e2e/`. Pull the exact acceptance
   wording from `docs/roadmap.md`'s "Acceptance" bullets for feature rows, and
   from the PLAN section 2 row's own description for tech-swap rows.
   Operationalize subjective PLAN language into a measurable proxy - e.g. "timebox
   drag feels native" (PLAN 5.3) becomes "drag completes, the new time persists
   after reload, zero console errors during the drag"; "60fps on a mid device"
   (PLAN Phase 7 gate) is NOT web-testable on this machine - rule it PARKED, do
   not fake a frame-rate assertion in headless Chromium and call it proof.
3. Run the staged validation ladder from a FRESH disposable PB dir - never reuse
   a dirty one; create the next unused
   `.scratch/pb-e2e/pb_data-<n>` per the orchestration framework's oracle
   section: `npm run typecheck` -> `npm run build` -> `npm run lint` -> boot
   smoke (`@l1`) -> auth (`@l3`) -> one write (`@l4`) -> `@l2`..`@l8` in order ->
   the full suite (`npx playwright test`, no `--grep` filter). Stop at the first
   REAL app defect: capture server logs + a screenshot + the Playwright trace
   BEFORE touching any code. A small, obviously in-scope fix (per IN
   SCOPE/OUT OF BOUNDS above) is fine and expected - that is what "full E2E
   parity" requires. Anything bigger - gap-rule that checklist row `BLOCKED` with
   the exact question, do not push a risky fix through just to turn the suite
   green.
4. Finish `docs/lynx-run/parity-checklist.md` with every Section A and Section B
   row ruled. Commit it alongside any new `e2e/*.spec.ts` files and any small
   in-scope fixes, together.

DONE MEANS (mechanical gate):
- `docs/lynx-run/parity-checklist.md` exists; every Section A row (7) and every
  Section B row (20, per PLAN section 2's table) has a Status.
- `npx playwright test` (the full suite, no `--grep` filter) is green, run
  against a freshly created disposable PB data dir (confirm in your CONTRACT
  which `.scratch/pb-e2e/pb_data-<n>` you used and that it was newly created,
  not reused).
- `npm run typecheck && npm run build && npm run lint` all green.
- Commit: `lynx(8.4-full-e2e-parity): parity checklist + closing e2e coverage`.
- If ANY row is ruled `BLOCKED`: this unit's CONTRACT status is "blocked", not
  "done" - say so plainly, and still commit whatever specs/checklist rows/fixes
  were safely completed.

PARKED (this machine cannot verify; do not fake):
- The "Native shell" row and any native-only proxy for "60fps on a mid device" /
  on-device font rendering / on-device safe-areas - all genuinely unverifiable
  on this Windows box with no Xcode/Android SDK; point these rows at
  `docs/lynx-run/native-hosts-runbook.md` (unit 8.5) rather than leaving them
  blank.
- Anything ruled `GAP - pre-existing` is NOT parked (parked = native-only,
  unverifiable HERE; gap = out of scope because it was never built at all) -
  keep the two concepts distinct in the checklist's Status column and in your
  CONTRACT.
