SPEC (concrete):

1) Theme mechanism on Lynx (crib: "no :root/html - use the preset's uiVariants for
   states"; "CSS variables ... supported"; FETCH https://lynxjs.org/rspeedy/styling and
   the styling guide and CITE the exact theming section in the result). The src-legacy
   app set tokens on `document.documentElement` (`:root`/`.dark` in global.css + inline
   overrides). Neither `document` nor `:root`/`html` exist on Lynx. Replace with a
   REACTIVE root view carrying the resolved CSS variables inline, which cascade to all
   descendants (CSS variables DO cascade on Lynx even though text properties do not
   inherit):
   - CREATE a `ThemeVars` component (in `src/features/style/`) that subscribes to
     `useThemeStore` (theme) + `useStyleStore` (colors/font/radius/grain/shadow) and
     renders a top-level `<view style={vars} class="...">` wrapping `<Outlet/>`-side
     content. Mount it at the app root so ALL screens are inside it (wrap the
     `rootRoute` component's children, or the app-root `<view>`). Confirm placement so
     both `<Backdrop/>` and routed content inherit the vars.
   - `vars` computation (port of `applyStyle` + `DEFAULT_COLORS`): let
     `mode = useThemeStore().theme`. For each `{key}` in `COLOR_TOKENS`, set
     `--${key}` = `styleColors[mode][key] ?? DEFAULT_COLORS[mode][key]` (ALWAYS emit a
     value - there is no `:root` fallback on Lynx, so the root view must carry the full
     palette, not just overrides). Set `--app-font-sans`/`--app-font-display` from
     `FONT_STACKS` (family names as registered by 2.1 - if 2.1 renamed families for the
     static-weight instances, use those names; record any mismatch), `--app-radius-card`
     = `${radius}rem`, `--grain-strength` = `String(grain)`, `--shadow-strength` =
     `String(shadow)`. Also emit `--radius-card`, `--soft-shadow`, and the grain PNG
     var per 2.1/2.2 if global.css moved them off `:root`.
   - global.css: the `:root`/`.dark` VALUE blocks move into this computed root-view
     style (or, fallback approach B: keep two class blocks `.theme-light`/`.theme-dark`
     in global.css and toggle the class on the root view). Keep the component classes
     (`.shadow-soft`, `.grain`, `.grain::after`, `.inset-well`, `.stamp-*`, `::selection`)
     and the Tailwind `@theme` token mapping - only the mode value source changes.
   - If the fetched styling doc says CSS variables do NOT cascade from an element to
     descendants on Lynx (contradicting the crib), STOP (this mechanism depends on it).
   - Light is the DEFAULT (CLAUDE.md "Theme is light + dark only. No system. Light is
     the default."): `useThemeStore` initial `theme: "light"`. No system-theme query.

2) Persistence via the storage adapter (PLAN 5.8; crib "localStorage does not exist"):
   - `src/stores/theme.ts`: keep `persist` but back it with the 1.4 storage adapter via
     `createJSONStorage(() => storageAdapter)` (name "dooey-theme"). Remove
     `applyTheme`/classList. `toggle`/`set` just update `theme` state (the `ThemeVars`
     view re-renders from it).
   - `src/features/style/store.ts`: keep `persist` (name "dooey-style", same
     `partialize`: colors, fontSans, fontDisplay, radius, grain, shadow, backdrop,
     dockDoodles) backed by the storage adapter. Remove `applyStyle` DOM writes and the
     `useThemeStore.subscribe(applyStyle)`. Keep `pageDoodles` NOT-persisted-locally
     (they live on the user record via `savePageDoodles`/`syncPageDoodlesFromUser` -
     preserve that PB sync verbatim; it uses `pb` from 1.4).
   - If the storage adapter is async and zustand `persist` hydration races the first
     `ThemeVars` render (a flash of default theme), that is acceptable (light default
     is the pre-hydration state); do NOT add a blocking gate unless the E2E "dark
     persists across reload" spec fails, in which case STOP and report (the gate
     belongs to the 1.4 seam).

3) Slider primitive (`src/components/ui/slider.tsx`) - the numeric control replacing
   `<input type=range>`. GESTURE-linked -> MTS (crib "Animation"; PLAN 5.3;
   https://lynxjs.org/react/main-thread-script - FETCH for exact `main-thread:bind*`
   touch event names + `event.currentTarget` geometry access). Props: `value`, `min`,
   `max`, `step`, `onChange(v)`, `label`, `disabled`. Implementation: a track `<view>`
   + a thumb `<view>`; a `main-thread` touch handler maps touch x within the track to
   a value, snaps to `step`, and updates the thumb via `setStyleProperty` for
   immediate feedback, forwarding the committed value to `onChange` (run on the
   background thread via the MTS->main bridge, e.g. `runOnBackground`, per the doc).
   MTS constraints: captured `min/max/step` must be JSON-serializable numbers (they
   are); no outer-scope mutation. If mapping touch geometry to a value is not
   expressible with the pinned MTS API, FALLBACK to a stepper (- / + buttons stepping
   by `step`, value shown) - record which was used; do NOT block the layer on slider
   polish.
