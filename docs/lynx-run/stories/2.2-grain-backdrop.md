TIER: normal

native platforms is out of mechanical reach here and is PARKED.

BROOM (old file/rule -> disposition):
- `--grain` (the inline `feTurbulence` SVG-filter data URI in `:root`): DROP. SVG filter
  (`<filter>`/`feTurbulence`) support on Lynx is unproven (PLAN 5.6) - replaced by a baked
  tiled PNG.
- `body::before` and `.grain::after` (pseudo-elements): DROP. Lynx has no pseudo-element
  support at all - "Lynx doesn't have pseudo-element support yet"
  (https://lynxjs.org/api/css/selectors.html). Replaced by a real `<view>` sibling element
  (`GrainOverlay`) positioned absolutely, since that is the only way to layer decorative,
  non-interactive content in Lynx.
- `background-attachment: fixed` (used on `.grain::after` so "a folder tab and its card
  sample the same texture" continuously across the viewport): DROP, cannot be ported as-is.
  Lynx's `<bg-layer>` explicitly "does not support the `<attachment>`" keyword
  (https://lynxjs.org/api/css/properties/background). Each grained surface now tiles from its
  own element origin instead of a shared viewport-locked origin - a small, deliberate, visible
  regression (adjacent grained panels may not sample identical noise) - recorded here, not
  silently dropped. Cheap to revisit later if Lynx adds `background-attachment` support.
- The old opacity values (`calc(0.7 * ...)` on `body::before`, `calc(0.5 * ...)` on
  `.grain::after`, `0.12` in dark mode): DROP the specific numbers, replaced by PLAN 5.6's
  explicit target of "2-3% opacity" - use `0.03` as the base for both the fixed wash and the
  per-panel `.grain-tile` class, in both light and dark mode (dark mode keeps the `overlay`
  blend mode, light mode keeps `multiply`, matching the old structure at the new number). The
  `--grain-strength` CSS var multiplier is kept (still `calc(0.03 * var(--grain-strength,
  1))`) so unit 3.4's Style Studio "grain strength" slider still has a live hook.
- `pngjs` (new devDependency): added SOLELY for `scripts/generate-grain.mjs`, a one-time /
  re-runnable offline generator - it is not imported by any shipped app code and does not
  appear in the web/native bundle. Pure-JS, no native bindings (safe on this Windows box,
  unlike `canvas`/`sharp`).

SPEC:

1. Add the generator dependency: `npm install -D pngjs` (and `npm install -D @types/pngjs` if
   the script is TypeScript; a plain `.mjs` script needs neither `@types` nor to be part of
   `tsc --noEmit`'s scope - keep it a `.mjs` file under `scripts/`, matching the existing
   `scripts/verify-program.mjs` convention).

2. Write `scripts/generate-grain.mjs`:
   ```js
   import { PNG } from "pngjs";
   import { writeFileSync } from "node:fs";

   // Deterministic seeded PRNG (mulberry32) so the tile is reproducible across runs.
   function mulberry32(seed) {
     return function () {
       seed |= 0;
       seed = (seed + 0x6d2b79f5) | 0;
       let t = Math.imul(seed ^ (seed >>> 15), 1 | seed);
       t = (t + Math.imul(t ^ (t >>> 7), 61 | t)) ^ t;
       return ((t ^ (t >>> 14)) >>> 0) / 4294967296;
     };
   }

   const SIZE = 160; // matches the old SVG filter's width/height
   const rand = mulberry32(42); // fixed seed - reproducible tile
   const png = new PNG({ width: SIZE, height: SIZE });

   // Independent per-pixel gray noise. Unlike smooth/Perlin noise, uncorrelated white
   // noise has no low-frequency structure to clash at tile edges, so a plain repeat is
   // already seamless - no edge-blending needed.
   for (let y = 0; y < SIZE; y++) {
     for (let x = 0; x < SIZE; x++) {
       const idx = (SIZE * y + x) << 2;
       const gray = 90 + Math.floor(rand() * 75); // 90-165: matches the old filter's look
       png.data[idx] = gray;
       png.data[idx + 1] = gray;
       png.data[idx + 2] = gray;
       png.data[idx + 3] = 255;
     }
   }

   writeFileSync("src/assets/grain.png", PNG.sync.write(png));
   console.log("wrote src/assets/grain.png (160x160)");
   ```

3. Run it once and check in the output: `node scripts/generate-grain.mjs`, then verify with
   `node -e "require('pngjs').PNG.sync.read(require('fs').readFileSync('src/assets/grain.png'))"`
   (must not throw) and confirm the file is committed (it is a real asset, not scratch).

5. Create `src/components/grain-overlay.tsx`:
   ```tsx
   import { cn } from "@/lib/cn";

   /** Real element replacing the old ::after paper-grain trick (Lynx has no
    * pseudo-elements: https://lynxjs.org/api/css/selectors.html). Absolutely fills
    * whatever positioned ancestor renders it; pass `className` to size/position it
    * (e.g. "absolute inset-0" inside a Panel, or "fixed inset-0 -z-10" for the
    * whole-screen wash). */
   export function GrainOverlay({ className }: { className?: string }) {
     return (
       <view aria-hidden className={cn("grain-tile pointer-events-none", className)} />
     );
   }
   ```
   (`<view>`, not `<div>` - Lynx elements per the crib sheet. `className` + Tailwind classes
   compile the same way as any other Lynx element per
   https://lynxjs.org/rspeedy/styling.)

6. Port `Backdrop.tsx` to `src/features/style/components/Backdrop.tsx`:
   ```tsx
   import { useStyleStore, type BackdropCrop } from "../store";

   export function Backdrop() {
     const url = useStyleStore((s) => s.backdropUrl);
     const settings = useStyleStore((s) => s.backdrop);
     if (!url) return null;

     const image = (crop: BackdropCrop) => (
       <image
         src={url}
         className="h-full w-full"
         style={{
           objectFit: "cover",
           objectPosition: `${crop.x}% ${crop.y}%`,
           transform: `scale(${crop.zoom * (1 + settings.blur / 300)})`,
           filter: settings.blur > 0 ? `blur(${settings.blur}px)` : undefined,
           opacity: settings.opacity,
         }}
       />
     );

     return (
       <view aria-hidden className="pointer-events-none fixed inset-0 -z-20 overflow-hidden">
         <view className="h-full w-full md:hidden">{image(settings.mobile)}</view>
         <view className="hidden h-full w-full md:block">{image(settings.desktop)}</view>
       </view>
     );
   }
   ```
   Notes: `<image>` replaces `<img>`; `object-fit`/`object-position` moved into the `style`
   object explicitly (Lynx `<image>` supports these as standard CSS properties per the crib's
   "CSS variables, gradients, and box-shadow are supported" - if `npm run build` reports any
   of `object-fit`/`object-position`/`filter`/`transform` as unrecognized on `<image>`, that is
   a genuine build-time signal, not a judgment call: record the exact rspeedy/lint error text
   in the unit's result and treat it as a blocked finding rather than silently dropping the
   property.
