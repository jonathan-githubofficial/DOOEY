# 1.5 - E2E oracle (Playwright + disposable PocketBase)
LAYER: L1-foundation / TIER: heavy / RISK: high

GOAL: Build the run's only success oracle - a @playwright/test harness that serves the
built Lynx web output against a DISPOSABLE PocketBase (never the operator's data) and
proves, in headless Chromium, that the app boots, a programmatic sign-in works, and a
record created via the PB API appears live in the app (SSE proof).

CONTEXT: Units 1.1-1.4 are done: the web build emits output, pb.ts resolves its host
from PUBLIC_PB_URL, the storage-backed authStore + useCollectionLive are wired, and the
entry renders a minimal live status surface (auth state + a live collection count).
This unit builds e2e/ and the @l1 spec. Every later layer ADDS specs tagged for its
layer; the L-gate is typecheck + build + that layer's tag + all previously-green tags.

IN SCOPE (files) - all NEW:
  e2e/README.md                 - exactly how servers start (operator-runnable)
  playwright.config.ts          - projects, webServer(s), globalSetup, @tag grep
  e2e/global-setup.ts           - launch disposable PB, seed superuser + test user,
                                  write .scratch/pb-e2e/creds.json, stash PB handle
  e2e/global-teardown.ts        - stop the PB process cleanly
  e2e/fixtures.ts               - test fixtures: a pb SDK client bound to the
                                  disposable instance, the test user's creds
  e2e/smoke.l1.spec.ts          - the @l1 smoke spec (boot + sign-in + live SSE)
  package.json                  - add "@playwright/test" (dev) + a "test:e2e" script
  .gitignore                    - ensure .scratch/ is ignored (it is) + playwright
                                  artifacts (test-results/, playwright-report/)

OUT OF BOUNDS:
- NEVER serve against pb/pb_data, never read/copy/delete it, never use the operator's
  .env.local creds. The disposable instance uses .scratch/pb-e2e/ ONLY.
- Do NOT modify pb_migrations or pb_hooks (HARD RULE). If a hook crashes the instance,
  that is a REPORTED FINDING, not a patch (SPEC 6).
- Do NOT build auth UI or router (L3). Sign-in here is PROGRAMMATIC (SPEC 4).

BROOM: nothing ported from src-legacy (the old app had no e2e suite). All new.

SPEC (numbered):

1. Disposable PocketBase - exact serve command (charter oracle):
     pb/pocketbase.exe serve --http 127.0.0.1:8091 \
       --dir .scratch/pb-e2e/pb_data \
       --migrationsDir pb/pb_migrations --hooksDir pb/pb_hooks
   - The data dir is BUILT FRESH from pb/pb_migrations on first serve. For a clean
     slate, create a NEW dir (.scratch/pb-e2e/pb_data-<n>) - never hand-delete
     (charter). global-setup picks a fresh dir path per run (or reuses if present and
     healthy - prefer fresh: timestamp/counter suffix).
   - Port is 8091 (NOT 8090 - that is the operator's dev PB; must not collide).
   - global-setup spawns this process (real binary, killable), waits for
     http://127.0.0.1:8091/api/health to return 200, then proceeds.

2. Seed a superuser NON-INTERACTIVELY, creds generated (never inlined literals in a
   committed file; generated at runtime into a gitignored file):
   - Generate a random email (e.g. e2e+<rand>@dooey.test) and a strong random
     password in global-setup.
   - Create/refresh the superuser idempotently against the disposable data dir:
       pb/pocketbase.exe superuser upsert <email> <password> --dir .scratch/pb-e2e/pb_data-<n>
     (PocketBase 0.39.x: `superuser upsert`; confirm with `pocketbase.exe superuser
     --help` if the subcommand name differs, and STOP-on-ambiguity if it does).
   - Write the generated superuser + test-user creds to .scratch/pb-e2e/creds.json
     (gitignored via .scratch). NEVER write to pb/pb_data, NEVER to .env.local.

3. Seed a TEST APP USER via the PB API (the users collection exists from
   pb_migrations/001_init.js):
   - In global-setup, using the pocketbase SDK pointed at http://127.0.0.1:8091,
     authenticate as the superuser (pb.collection("_superusers").authWithPassword),
     then create the app user:
       pb.collection("users").create({ email, password, passwordConfirm: password })
     Set any fields the users collection requires; if create fails on a required
     field or on verification, use the superuser to update the record
     (verified: true, timezone, etc.). Do NOT edit the migration to relax rules -
     satisfy the schema from the client. Save these creds into creds.json too.
   - This user owns every record the specs create (owner = user id), matching the
     app's data-isolation rules.

4. The @l1 smoke spec (e2e/smoke.l1.spec.ts), tagged @l1 (via test title suffix
   " @l1" or test.describe tag so `npx playwright test --grep @l1` selects it):
   a) BOOT: navigate to the served web output; assert the status surface renders
      (the known <text> string / data attribute from unit 1.4). Signed-out state
      shows first.
   b) PROGRAMMATIC SIGN-IN: obtain a valid auth token for the test user WITHOUT a
      login UI. Preferred: in the browser context, run the app's own sign-in through
      page.evaluate against the app's pb instance, OR seed the storage-backed authStore
      key (appStorage "pb_auth") via page.addInitScript with a token obtained
      Node-side (pb.collection("users").authWithPassword) and reload so initSession
      validates it. Assert the surface flips to the signed-in state (this also proves
      the 1.4 storage-backed authStore + initSession definitive-4xx rule on a VALID
      session).
   c) LIVE SSE PROOF: read the live collection count on the surface; then Node-side
      (as the test user) create a record in the subscribed collection
      (pb.collection("tasks").create({ owner, title, ... })); assert the surface's
      live count increments WITHOUT a manual reload (useCollectionLive + the browser
      EventSource delivering the realtime event). Use Playwright expect.poll with a
      short timeout. This is the SSE round-trip proof PLAN 5.4 called the key spike.

5. Serving the built web output. Primary (mirrors production, no CORS): place the
   built web output where the disposable PocketBase serves static files so the app is
   SAME-ORIGIN with PB, and navigate to http://127.0.0.1:8091. Confirm PocketBase's
   static dir/flag via `pocketbase.exe serve --help` (it serves a public dir - default
   `pb_public` next to the data dir; use the flag if one exists). Fallback (if the
   public-dir mechanism is unclear): build with PUBLIC_PB_URL=http://127.0.0.1:8091
   (import.meta.env is compile-time) and serve the web output from a static server via
   playwright webServer (e.g. a small node http server on 127.0.0.1:4173), relying on
   PocketBase's default permissive CORS. Document whichever you use in e2e/README.md.
   The gate assumes `npm run build` already ran (staged ladder: build precedes e2e);
   the config may also run build in webServer.command to be self-contained.

6. calendar-sync.js soft-fail check (report, do NOT patch - HARD RULE). The hook at
   pb/pb_hooks/calendar-sync.js registers a */10min cron that reads calendar_links and
   calls Google. With NO Google env (CALENDAR_ENC_KEY / GOOGLE_CLIENT_ID / _SECRET)
   and no calendar_links rows, it should no-op (empty findAllRecords) or throw INSIDE
   the caught per-link loop / logged cron error - the disposable PB must STAY UP and
   keep answering /api/health. In global-setup (or a tiny probe test), after serving
   with --hooksDir pb/pb_hooks, confirm the instance is healthy and stays healthy.
   If loading the hook or the first cron tick CRASHES PocketBase, record it as a
   FINDING in the unit result (message + exit code + relevant PB log lines) and mark
   the unit blocked for a human ruling - do NOT modify pb_hooks to work around it.

7. e2e/README.md must document, operator-runnable and exactly:
   - the disposable PB serve command (SPEC 1), the 8091 port choice, and that
     .scratch/pb-e2e/ is the ONLY mutable dir (pb/pb_data is untouchable);
   - how creds are generated into .scratch/pb-e2e/creds.json (and that they are never
     the operator's .env.local);
   - how the web output is served (SPEC 5) and how to run:
       npm run build
       npx playwright test --grep @l1
   - how to reset (stop, make a NEW .scratch/pb-e2e/pb_data-<n>, never hand-delete).

DONE MEANS (exact commands + expected outcomes):
  1) npm run typecheck -> exits 0 (e2e/ is in the tsconfig include).
  2) npm run lint      -> exits 0.
  3) npm run build     -> exits 0 (web output emitted).
  4) npx playwright test --grep @l1  -> 1 spec, PASS: app boots, programmatic
     sign-in flips the surface to signed-in, and a PB-API-created record raises the
     live count via SSE. (First run may need `npx -y playwright install chromium`.)
  5) The disposable PB ran on 127.0.0.1:8091 against .scratch/pb-e2e/pb_data-<n>;
     pb/pb_data was never touched (verify: its mtime unchanged; no serve pointed at it).
  6) global-teardown stopped the PB process (no orphan on 8091).
  7) calendar-sync.js soft-fail confirmed (PB healthy throughout) OR a crash is
     reported as a finding (unit blocked, pb_hooks unmodified).
  @l1 is now the first live tag. Commit: "lynx(1.5): E2E oracle - Playwright + disposable PB, @l1 smoke".
