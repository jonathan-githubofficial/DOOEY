Two platform facts, verified against lynxjs.org while writing this story (2026-07),
drive several decisions below - read them before writing code:
1. Lynx's `<input>` element supports only `type` in {text, number, digit, password,
   tel, email} (https://lynxjs.org/api/elements/built-in/input.html). There is no
   file-picker type on ANY Lynx target (native or web) - this is not a web-only gap.
2. `<text>` does not inherit CSS from ancestor `<view>` nodes, but a nested `<text>`
   INSIDE a `<text>` DOES inherit `color`/`font-family` from its parent `<text>` (and
   can override them) - useful for inline formatting later, and relevant here for
   truncation: use the `text-maxline` attribute on `<text>` (not Tailwind's
   `line-clamp-*`, which relies on a webkit-only CSS property the Lynx preset does not
   carry) wherever src-legacy used `line-clamp-2`.
   (https://lynxjs.org/api/elements/built-in/text.html)

3. Port FolderCard.tsx (src/features/learning/components/FolderCard.tsx). Element and
   behavior mapping, decided now so no judgment call is needed at build time:
   - Root `motion.div` (role="button") -> Lynx `<view>` with a tap binding. Use
     WHATEVER tap/press-feedback mechanism unit 2.3 (StampButton's
     `active:scale-95`-equivalent) or unit 4.2 (Check's press-depress) already
     established for "this whole element responds to a tap with a small depress" -
     mirror that exact mechanism here; do not invent a second one. If the two
     disagree, prefer StampButton's (it is the earlier, more foundational primitive).
     This satisfies CLAUDE.md's "press states depress (scale/shadow)" rule via CSS
     transition, not Main-Thread Scripting (this is a discrete tap, not a
     gesture-linked drag - PLAN 5.3's "declarative enter/exit/settle" bucket, not the
     MTS bucket).
   - DROP the `whileHover="lift"` peeking-papers animation entirely. Lynx targets
     touch first; there is no hover concept, and the crib sheet is explicit: "No
     pseudo/data-attr selectors - use the preset's uiVariants for states"
     (https://lynxjs.org/rspeedy/styling). Render both peeking-paper `<view>`s
     PERMANENTLY at their old `rest` transform values (`y:0 rotate:-2deg` for the left
     one, `y:3 rotate:2deg` for the right one) - a static resting look, not a
     hover-triggered one. Use plain CSS `transform` in the style object; no
     `motion/react` import in the ported file.
   - `<span>`/`<p>` text nodes -> Lynx `<text>`, each with its OWN explicit
     `color`/`font-family`/`font-size` (text does not inherit - fact 2 above). Anywhere
     the old code used Tailwind `line-clamp-2` (the goal title), replace it with the
     Lynx `<text text-maxline={2}>` attribute instead of a CSS class.
   - The dress-up "Paintbrush" button's `opacity-0 group-hover:opacity-100` reveal ->
     DROP the hover-reveal (same "no hover" reasoning as above). Show it always at a
     reduced but tappable opacity (e.g. the old `hover:opacity-100` end-state's
     visual weight) so it is discoverable and tappable on a touch screen without a
     hover trigger.
   - `<img>` (cover picture) -> Lynx `<image src={cover} />`. FolderCard only ever
     READS `program.cover` here (uploading a new one is FolderStudio's job, and that
     upload path is dropped per BROOM) - keep this read-only render, it costs nothing
     and Lynx `<image>` supports remote URLs the same way `<img src>` did.
   - `DoodleSvg` usage (existing saved doodle) -> unchanged import, reused from
     unit 3.3 as-is.
   - `AnimatePresence`/`<FolderStudio>` mount-on-demand -> keep the `studio` boolean
     state and conditional render; drop `AnimatePresence` (no exit-animation library
     exists on Lynx). If an earlier unit (2.x/3.x/4.x) already built a shared modal
     enter/exit helper, reuse it for FolderStudio's overlay; if none exists yet,
     enter-animate only (fade/scale in on mount via a CSS transition) and skip an
     exit animation on close (the overlay just unmounts) - an intentional, documented
     simplification, not a blocking defect.
   - Everything computing colors (`hsl(var(...))`, `color-mix(...)`, linear-gradient
