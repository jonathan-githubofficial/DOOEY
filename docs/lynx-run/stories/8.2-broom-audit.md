# 8.2 - Final broom audit

LAYER / TIER / RISK: L8-cutover / normal (sonnet) / MEDIUM-HIGH (this unit has
real deletion authority over whatever is left in src-legacy/, plus a dead-code
sweep touching the live src/ tree and package.json - both can break the build if
done carelessly; the hard-rule STOP-on-genuinely-undecidable-deletion applies).

GOAL: Empty and remove `src-legacy/` for good, write the consolidated broom
record for the whole run, rule on every file still left in it, sweep the NEW
tree for dead code/deps, and leave typecheck+build+lint green.

CONTEXT (what exists when this unit starts):
- L1-L7 are committed and E2E-green through @l7. Unit 8.1 (earlier in this same
  layer) already rewired the Docker web-build stage - not this unit's concern.
- Per the tree convention, every unit 1.1-7.3 ported what its own story listed
  out of `src-legacy/` into the new `src/` tree, then `git rm`'d the
  `src-legacy/` files it consumed (ported OR deliberately dropped - each unit's
  own story records which, in that story's BROOM section). `src-legacy/` should
  therefore be small or already empty by the time this unit starts - this unit is
  the final backstop, not the primary broom (do not re-litigate a decision an
  earlier unit's BROOM section already made and a `git log` commit already
  executed - record it, don't redo it).
- `docs/lynx-run/run-manifest.json` is written by the harness's Digest phase,
  which runs AFTER the entire L8 layer (including this unit) completes. It will
  almost certainly NOT exist yet when this unit runs - that is expected, not an
  error. Do not wait for it and do not treat its absence as blocking. If it
  happens to already exist (e.g. a prior partial run left one from a resume),
  fold in its per-unit `broom`/`parked` fields as a bonus source, but the primary
  sources of truth are:
  (a) every story file `docs/lynx-run/stories/1.1-*.md` through `7.3-*.md` - each
      has a mandatory BROOM section per the orchestration framework's template;
  (b) `git log` on `lynx/migration` - the actual `git rm` / file changes each
      unit's commit made. A story can SAY a file was dropped; only `git log`
      proves it happened.

IN SCOPE (files):
- `src-legacy/` - everything still present: read, rule, `git rm`.
- `docs/lynx-run/broom-record.md` - NEW file, this unit's primary written
  deliverable.
- `src/` - READ for the dead-code sweep (unused exports/imports); EDIT only to
  delete genuinely dead code found by the sweep (no refactors, no feature
  changes, no fixes of pre-existing bugs unrelated to dead code).
- `package.json` - dependencies/devDependencies dead-package sweep.
- `docs/lynx-run/stories/*.md` (1.1 through 7.3) - READ ONLY, source of truth for
  each unit's BROOM section.
- `docs/lynx-run/run-manifest.json` - READ ONLY IF present (see CONTEXT; its
  absence is normal).
- `git log` on `lynx/migration` - READ ONLY.

OUT OF BOUNDS:
- `src/` application code changes beyond deleting provably-dead
  exports/imports/files found by the sweep.
- `Dockerfile` / compose / deploy.yml (8.1's territory - already done).
- `CLAUDE.md` and the rest of the docs sweep (8.3's territory).
- E2E specs / the parity checklist (8.4).
- The native runbook (8.5).
- `pb/`, `pb_migrations`, `pb_hooks`, `pb/pb_data` - never touch.
- Re-deciding a broom call an earlier unit's story already made and a
  corresponding `git rm` commit already executed - record it in the consolidated
  record, do not touch those files again.

BROOM (per old file: port / drop / defer) - this unit's entire job is running
this decision procedure over whatever remains in `src-legacy/`:
1. **PORT LATE** - the file implements in-scope product behavior (matches a
   CLAUDE.md "Current scope (locked feature list)" item, or a row of PLAN
   section 2's before/after table) that no unit's IN SCOPE ever claimed - it was
   missed by the layering. Port it now, following the same pattern (adapt vs.
   rewrite, per PLAN section 4's bucket for that kind of file) the owning layer
   would have used; add it to `src/`; then `git rm` the `src-legacy` copy. Record
   as "ported at 8.2 (missed by earlier layering)" with the reason.
2. **DROP WITH REASON** - dead code, a superseded experiment, a duplicate, or
   something CLAUDE.md's no-dead-code rule / PLAN section 4's "port with a
   broom" preamble already argues against keeping (the PLAN is explicit: "Prefer
   deleting to porting. ... When in doubt about dropping something non-trivial,
   flag it rather than silently porting it"). `git rm` it. Record the file plus a
   one-line reason.
3. **BLOCKED (genuinely undecidable)** - you cannot tell from the story files,
   `git log`, CLAUDE.md, or the PLAN whether this should be ported or dropped
   (e.g. it looks like real product logic but no locked-scope feature obviously
   owns it, or a decision would touch auth/session semantics or a data-shape
   question the orchestration framework's hard rules reserve for a human). Do
   NOT `git rm` it. List it in `broom-record.md`'s "BLOCKED - needs operator
   ruling" section with the exact question. This is the ONLY path where
   `src-legacy` is allowed to still contain a file when this unit finishes - but
   this unit's own CONTRACT status is then "blocked", not "done", per the hard
   rule: "On ambiguity ... or any HIGH-RISK call ... deleting something no story
   brooms out, STOP". Do not improvise past a genuine BLOCKED.

SPEC (numbered, concrete):
1. Consolidate the record: for every `docs/lynx-run/stories/{1.1..7.3}-*.md`
   file, extract its BROOM section verbatim into `broom-record.md` under a
   heading per unit. Cross-check each against `git log --name-status` for that
   unit's commit (`git log --all --grep '^lynx(1\.1'` etc., or filter the full
   log by commit subject) to confirm the stated broom decisions actually
   happened - a `git rm` in the diff proves it; a story claim with no matching
   `git rm` is a mismatch. Note any story/commit mismatch as its own finding line
   (do not silently trust either source alone; do not "fix" the mismatch by
   deleting something retroactively unless step 2's procedure below independently
   rules DROP on it).
2. `git ls-files src-legacy/` to list everything still physically present. For
   each remaining file or directory, apply the three-way ruling above (PORT LATE
   / DROP WITH REASON / BLOCKED). Work file-by-file, not by vibes - if a
   directory is entirely leftover test fixtures or a duplicate of something
   already ported verbatim, you may rule the directory as one line, but say so
   explicitly (do not silently skip files inside it).
3. After every file is ruled PORT LATE or DROP (the target is zero BLOCKED, but
   an honest BLOCKED beats a wrong guess - see hard rules), confirm
   `git ls-files src-legacy/` returns nothing, then `git rm -r src-legacy` (a
   no-op if already empty) and confirm the directory is gone from the working
   tree (`test -d src-legacy` fails).
4. Write `docs/lynx-run/broom-record.md`:
   - One section per unit 1.1-7.3: its BROOM summary + a "matches git log:
     yes/no" line.
   - A final "## 8.2 final sweep" section: every file ruled at this step, with
     its disposition (PORT LATE / DROP / BLOCKED) and one-line reason.
   - A final "## Dead-code sweep" section (see step 5).
   Pure ASCII, terse bullet lines - no prose walls, this is a record, not an
   essay.
5. Dead-code sweep of the NEW `src/` tree (CLAUDE.md's "No dead code" rule,
   verified now rather than left to drift as 96 files' worth of ported/rewritten
   code accumulated over 7 layers): run a real unused-export/import detector
   non-interactively - prefer `npx -y knip` (it also flags unused
   `package.json` dependencies in one pass); if its default config fights the
   Lynx project shape (e.g. it does not understand `.lynx.bundle` entries or
   rspeedy's config format) and getting it working cleanly is more effort than
   it is worth, fall back to `npx -y ts-prune` for exports only and check
   `package.json` deps by hand (`grep -rl "from ['\"]<pkg>" src/` per dependency).
   For every genuinely unused export/import/file flagged that is NOT a false
   positive - barrel-file re-exports, framework entry points (`main.tsx`,
   `App.tsx`), anything a route or a lazy import references dynamically (grep
   before deleting) - delete it. Cross-check `package.json` against PLAN section
   8's explicit remove-list (`@capacitor/*`, `motion`, `tw-animate-css`,
   `tailwindcss@4`-era entries, `@tailwindcss/vite`, `@fontsource-variable/*`,
   `vite`, `@vitejs/plugin-react`, `@radix-ui/react-slot`) - if any of these
   somehow survived past 1.6/2.x, remove them now and record it; also remove any
   OTHER genuinely-unused dependency the sweep tool finds that PLAN section 8
   did not anticipate. Record every removal (file or dependency) in
   `broom-record.md`'s "Dead-code sweep" section with what was removed and why.
6. Re-run `npm run typecheck && npm run build && npm run lint` after EACH
   deletion batch, not only at the very end - a dead-code tool is a hint, not an
   oracle. If removing something breaks the build (a dynamic import, a route
   file the tool could not trace, a type-only import), put it back immediately
   and note it in the sweep section as a confirmed false positive with why.

DONE MEANS (mechanical gate):
- `git ls-files src-legacy/` -> empty output.
- `test -d src-legacy` -> fails (directory does not exist).
- `docs/lynx-run/broom-record.md` exists and covers every unit 1.1-7.3 plus the
  "8.2 final sweep" and "Dead-code sweep" sections, with zero-or-more BLOCKED
  items each carrying an exact operator question.
- `npm run typecheck`, `npm run build`, `npm run lint` all green.
- If ANY file was ruled BLOCKED: this unit's CONTRACT status is "blocked", not
  "done" - say so plainly, and still commit whatever was safely decided (the
  PORT LATE / DROP files actually removed) so the next attempt does not redo
  settled work.
- Commit (only if status is done/noop):
  `lynx(8.2-broom-audit): final src-legacy sweep + dead-code pass`.

PARKED (this machine cannot verify; do not fake):
- None expected - this is a pure repo-hygiene and documentation unit. A BLOCKED
  ruling on an undecidable file is NOT a "parked" item (parked = native-only,
  unverifiable on this machine); it is a "blocked" unit status requiring an
  operator decision. Keep the two concepts distinct in the CONTRACT - do not
  report a BLOCKED file under PARKED.
