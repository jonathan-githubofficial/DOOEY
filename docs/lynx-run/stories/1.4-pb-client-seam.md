SPEC (numbered):

1. Host resolution via rspeedy env (replacing window.location/import.meta.env-Vite).
   Rsbuild/Rspeedy exposes: import.meta.env.MODE / .DEV / .PROD, and custom vars must
   be prefixed PUBLIC_ (e.g. import.meta.env.PUBLIC_PB_URL, set via .env files or
   `source.define` in lynx.config.ts). Resolution order (mirrors the old intent,
   CLAUDE.md "Deployment"):
     const host =
       import.meta.env.PUBLIC_PB_URL            // explicit (native builds, e2e)
       || (import.meta.env.DEV ? "http://127.0.0.1:8090"
            : (typeof window !== "undefined" ? window.location.origin : ""));
   - Web same-origin uses window.location.origin (present on the web target; the
     Docker image serves the app from PocketBase, so same-origin is correct - CLAUDE.md).
   - NATIVE has no window.location: it REQUIRES PUBLIC_PB_URL. PARK the native default;
     document that a native build must set PUBLIC_PB_URL.
   Doc: Rsbuild environment variables (PUBLIC_ prefix, import.meta.env.DEV/PROD/MODE).
   STOP-on-ambiguity: if PUBLIC_-prefixed import.meta.env vars are not exposed by this
   rspeedy version, read the rsbuild env doc and use source.define instead; if still
   unclear, return "blocked" with the doc findings.

2. Fetch adaptation. The pocketbase SDK uses global fetch. On the WEB target the
   browser fetch is used automatically (no change needed). On NATIVE, Lynx provides
   its own fetch; standard streaming is experimental (enableFetchAPIStandardStreaming)
   - PARK that. Do NOT inject a custom fetch for web unless the SDK requires it; keep
   the default. Doc: https://lynxjs.org/guide/interaction/networking .

3. Realtime (SSE). pb.realtime uses EventSource. On WEB the browser EventSource works
   (this is what the 1.5 SSE proof exercises). For NATIVE, wire lynx.EventSource so
   the SDK uses it - per PLAN 4b/5.4 the SDK is "wired to Lynx fetch + lynx.EventSource
   (same API as web)". Provide lynx.EventSource to the SDK's realtime layer only when
   running on native (guard on the web/native environment); on web leave the global
   EventSource. PrimJS lacks TextEncoder/TextDecoder for SSE payload decoding on
   NATIVE (a TextDecoder shim / TextCodecHelper may be needed) - PARK the native SSE
   decode; the web target uses the browser's TextDecoder. Doc: PLAN 5.4,
   https://lynxjs.org/guide/interaction/networking , crib sheet.
   STOP-on-ambiguity: if the pocketbase SDK version does not expose a way to inject
   the EventSource implementation, note it as a finding; the WEB path (browser
   EventSource) still satisfies the 1.5 gate, so do not block on the native injection.

4. Storage adapter + authStore (HIGH-RISK - session semantics). Back pb.authStore
   with the KV seam so auth persistence does not depend on the browser localStorage
   global:
   - Use the pocketbase SDK's AsyncAuthStore, backed by appStorage:
       import PocketBase, { AsyncAuthStore } from "pocketbase";
       const store = new AsyncAuthStore({
         save:    (serialized) => appStorage.setItem("pb_auth", serialized),
         initial: appStorage.getItem("pb_auth"),
         clear:   () => appStorage.removeItem("pb_auth"),
       });
       export const pb = new PocketBase(host, store);
     On web, appStorage is localStorage-backed, so the session round-trips exactly as
     before. On native, appStorage is the native KV (PARKED).
   - PRESERVE the CLAUDE.md "Auth & sessions" contract EXACTLY, end to end:
       * initSession() (features/auth/api.ts) refreshes the token/record on boot and
         drops the session ONLY on a definitive 4xx (`ClientResponseError` with
         status >= 400 && < 500). Network failures and 5xx must NEVER clear the
         session. Verify the ported initSession still does exactly this - do not
         "improve" it. This is the single most load-bearing auth rule in the app.
       * pb.authStore.onChange (in stores/auth.ts and later router.invalidate at 3.1)
         keeps the store in sync; do not change its shape.
   - Any change to this session rule is a STOP (charter: auth/session semantics are
     HIGH-RISK). If AsyncAuthStore's async getItem timing would change initSession's
     boot behaviour, note it and keep the definitive-4xx-only drop rule intact.
