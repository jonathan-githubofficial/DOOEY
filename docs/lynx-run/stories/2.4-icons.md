SPEC:

1. Resolve the `lucide-react` question with a real build + screenshot, not a guess (PLAN 5.9:
   "Validate in the spike" - this unit IS that validation, since no prior spike note exists):
   a. In `src/pages/Gallery.tsx` (from unit 2.3), temporarily add:
      `import { Plus } from "lucide-react";` and render `<Plus className="text-ink" />`
      somewhere inside a `data-testid="gallery-icon-smoketest"` wrapper `<view>`.
   b. `npm run build` (web target). If the build fails (rspack/rspeedy rejects the unknown
      host tags `path`/`svg` that `lucide-react`'s compiled output emits, or a TS/JSX error),
      that is decision B (fallback) - stop here, remove the temporary import, go to step 2.
   c. If the build succeeds, serve the web output and use Playwright to load it, then:
      `page.locator('[data-testid="gallery-icon-smoketest"] svg').count()` must be `1`, and
      `page.locator('[data-testid="gallery-icon-smoketest"] path').count()` must be `>= 1`
      (lucide's `Plus` icon is two `<path>`/`<line>` primitives). Also screenshot it
      (`page.locator('[data-testid="gallery-icon-smoketest"]').screenshot(...)` to
      `.scratch/2.4-lucide-smoketest.png`) and visually confirm a plus-sign shape, not a blank
      box (an element can exist in the DOM while rendering nothing if Lynx's web-elements host
      silently drops unrecognized children).
   d. If BOTH the count checks pass AND the screenshot shows a real plus sign: decision A
      (keep `lucide-react`). Remove the temporary smoketest markup from `Gallery.tsx` (the real
      Icons section in step 3 replaces it). If either check fails: decision B (fallback), go to
      step 2.
   e. Whichever way it goes, record the finding as a short note in this unit's commit message
      (e.g. "lucide-3.7-svg spike: build succeeded, count checks passed, screenshot confirmed -
      keeping lucide-react" or "...build failed with <exact error> - falling back to inline
      icons") - this becomes the spike-findings note the PLAN expected to already exist.

2. IF decision A (lucide-react validated for web):
   - Keep the `lucide-react` dependency. Create `src/components/icons/lucide.ts`:
     ```ts
     export {
       Check, Eraser, Pencil, Undo2, X, NotebookPen, CalendarDays, FolderOpen, Shapes,
       UserRound, ArrowLeft, Trash2, Loader2, Paperclip, Plus, ChevronLeft, ChevronRight,
       Minus, Link2, ListChecks, StickyNote, Maximize2, Palette, LogOut, Moon, Sun,
       ArrowUpRight, Clock, ExternalLink, RotateCw, ImagePlus, RotateCcw, ChevronUp,
       MoreHorizontal, Paintbrush, Upload, ArrowUpFromLine, Download, ChevronDown, BookOpen,
       Dumbbell, GraduationCap, Languages,
       type LucideIcon,
     } from "lucide-react";
     ```
     This barrel is the one import path every later unit's consumer file should use going
     forward (`import { Trash2 } from "@/components/icons/lucide"` instead of `from
     "lucide-react"` directly) - centralizing it here means if native later proves
     `lucide-react` does NOT work there (a real risk, since this smoke test only covers web),
     swapping the fallback in later only touches this one file, not 33 consumers.
   - Native risk note (record verbatim in the unit result, cite PLAN 5.9): `lucide-react`
     emits plain React `<svg>`/`<path>` elements; this test only proves the WEB output (which
     runs inside an actual browser via `@lynx-js/web-core`) renders them. Native hosts render
     through Lynx's own engine, where the SVG element "offloads parsing to a background thread
     and renders the entire graphic as a single native view" - whether nested JSX children
     compile the same way through the NATIVE Lynx compiler (vs. the web target's browser
     passthrough) is unconfirmed and cannot be tested on this machine (no Xcode/Android SDK).
     Unit 8.5 must re-run an equivalent smoke test on a real device/simulator before shipping
     native builds; if it fails there, fall back to the inline-`<svg>` approach in step 3 for
     native only (the barrel file makes that a one-file swap).

3. IF decision B (fallback - hand-author inline Lynx `<svg>` icons):
   - For each of the 42 icons in the manifest, extract its exact child-element data from the
     installed package (mechanical, not from memory - lucide ships each icon's raw element
     tree as source): read `node_modules/lucide-react/dist/esm/icons/<kebab-case-name>.js`
     (e.g. `plus.js`, `chevron-left.js`, `arrow-up-from-line.js` - kebab-case the PascalCase
     name) - it exports an array of `["tag", { ...attrs }]` tuples (e.g. `["path", { d: "M5
     12h14" }], ["path", { d: "M12 5v14" }]` for `Plus`). Transcribe those tuples verbatim as
     nested Lynx-legal SVG child tags (Lynx's `<svg>` supports exactly these as nested
     children, confirmed 17-tag list including `path`/`circle`/`rect`/`line`/`polyline`/
     `polygon`: https://lynxjs.org/next/blog/lynx-3-7 element reference) inside a 24x24
     `<svg>` wrapper matching lucide's own default presentation attributes (`viewBox="0 0 24
     24"`, `fill="none"`, `stroke="currentColor"`, `strokeWidth={2}`, `strokeLinecap="round"`,
     `strokeLinejoin="round"`).
   - One file per icon under `src/components/icons/lucide/`, kebab-case filename, matching the
     existing ornament pattern (`Squiggle.tsx` is one file, one component) - e.g.
